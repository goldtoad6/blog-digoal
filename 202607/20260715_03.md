## 背景  
最近我发现一个问题, 就是不上班之后, 我的“工作”时间明显比上班期间更长了.  
  
原因是 AI 越来越强, 我几乎沉浸在使用 AI 的过程中无法自拔.  
  
我觉得不能再这样下去, 索性分析了一下我使用 AI 的方式, 似乎是我自己的问题.  
  
简单来说就一句话: 我还没有摆脱“打工者”或者说“优秀打工者”的思维, 我写了一堆的 Skill, 然后我在 Claude CLI 中自己调用这些 Skill 干活.  
  
我还是个“打工者”, 甚至由于 AI 的强悍, 它不停下来, 我就不停下来, 我这个“打工者”的休息时间更少了.  
  
我其实是被 AI 击穿了, 就像双11时, 如果没有队列、没有缓存, 那么高的并发下, 数据库被击穿一样.  
  
那怎么办?  
  
我得转变思维, 转变成老板思维或者管理者思维.  
  
你见过哪个老板会亲自干活的? 没有, 都是吩咐下属, 下属如果也是个管理者, 他会继续吩咐下属.  
  
我得从直接调用 Skill , 转变成调用 Agent , 让 Agent 去调用 subagent 或 Skill 干活.  
  
我需要做的事情是: 定义 AI 给我的最终交付物的“可验证规范”、任务过程中间物料产出及“自验证”、以及在前面提到的约束条件成立的情况下对 AI 的“信任”.  
  
## Claude 设计哲学  
在纠正我的思维模式之后, 我要先理解 Claude 的设计哲学, 这样才能更好的用好这个工具.  
  
先说我的理解, 然后让 Claude 给我纠偏.  
  
skill: 代表各个专业场景的技能  
  
agent: 代表各专业方向的员工  
  
底层优秀员工思维: 写一堆 skill , 自己在 Claude CLI 主会话中用 skill 完成目标  
  
优秀管理者思维: 写一堆 agent , 通过编排 , 驾驭它们干活完成目标 (后面会说, 我这么理解是错的)   
  
不知道我的理解对不对?  
  
-----  
  
### 纠偏：skill 和 agent 不是"员工能力 vs 管理者能力"的对立，而是两个正交维度  
  
- **Skill = "怎么做"（how）** ：一份可复用的操作手册/流程知识，本身不占用独立上下文，只在被触发时把内容注入当前会话。  
- **Agent/Subagent = "谁来做、在哪个上下文里做"（who / where）** ：一个拥有独立上下文窗口、可以独立跑一段任务、只把最终结果返回给主会话的执行体。  
  
所以更准确的类比不是"底层员工 vs 管理者"，而是：  
  
> Skill 是知识/SOP 文档，Agent 是执行这份 SOP 的"人"（或者说，一个隔离的执行环境）。  
  
一个 agent 完全可以在执行过程中调用多个 skill；一个 skill 也完全可以不经过任何 agent，直接在主会话里被 Claude 自己读取执行。  
  
真正对应"管理者驾驭多个执行者"这个直觉的，其实是:  
- **subagent 编排**：主 Claude 把任务拆解，分发给多个 subagent 并行执行（各自独立上下文），再汇总结果给 Claude 主会话或人类。  
  
"底层员工自己完成目标"对应的是:  
- Claude CLI 主会话直接加载一堆 skill，人类自己选择调用哪些 skill 执行。  
  
用 agent 还是 skill? 判断标准：  
- 需要**独立上下文/并行/隔离**（比如长代码审查不想污染主上下文、多路探索互不依赖）→ 用 subagent  
- 需要**可复用的流程知识**，会反复被触发 → 用 skill  
- 以上两者都要 → 组合：subagent 内部调用 skill（这是目前公认最强的模式，一个 subagent 加载专属 skill 完成方法论，再通过 MCP 拿数据，返回总结给主会话）  
  
### 实践建议  
  
我最近写了一些 PostgreSQL SKILL : https://github.com/digoal/skills/tree/main/postgresql  
  
`find-postgres-bug`、`pg-deploy-cluster`、`pg-runtime-risk`、`pg-sql-audit`、`pg-host-resource-risk` 等, 本质上都是**纯 skill 层** —— 统一 read-only、`PGPASSWORD`、中文输出、多文件拆分 500 行内。这套是"知识层"做得非常扎实。  
  
如果要往"编排"方向进化，可以考虑的方向：  
1. 把重量级、长耗时、需要独立诊断上下文的环节（比如 `pg-load-spike-forensics` 的九步根因分析、`pg-bloat-root-cause` 的 MVCC 因果链推导）拆成 subagent 执行，只把结论摘要返回主会话 —— 避免几十个 SQL 查询结果把主上下文撑爆。  
2. 用一个"调度型" skill/命令做总入口（比如 `pg-health-checkup`），根据用户描述的症状，决定要唤起哪几个专项 skill（`pg-find-bloat` + `pg-find-unused-index` + `pg-parameter-tuning-advisor`），这是纯 skill 层面的编排，不需要 subagent 也能做。  
3. 真正需要多个"视角"并行独立分析、互不干扰、最后汇总的场景（这个在我的 `multi-expert-analyzer` SKILL 里其实已经在用了），才是 subagent 并行编排最有价值的地方。  
  
### 最佳实践清单  
  
**Skill 设计实践：**  
- description 字段是唯一被 Claude 常驻扫描的部分，写清楚"做什么 + 何时用"，这是触发准确率的关键。  
- SKILL.md 正文保持精炼 —— 一旦加载，内容会常驻上下文并在后续每轮都计入 token 成本，把"为什么这样设计"这类解释性内容挪到 skill 的附属文件，正文只留"做什么"。  
- 复杂技能拆成多文件（主 SKILL.md + examples.md + scripts/），保持主文件聚焦，这也是你现在遵循的模式。  
- 明确 `disable-model-invocation` 还是允许模型自主调用 —— 如果你也因为 SKILL 太多而踩过 `skillOverrides` 的坑，在 SKILL.md frontmatter 里显式声明比全局配置更可靠。  参考 [《Claude , Codex 使用经验总结》](../202604/20260415_05.md)  
- 一次性、不会复用的任务（比如某次特定的数据库迁移）不要写成 skill，直接内联给指令；只有会反复出现的模式才值得沉淀成 skill。  
  
**Subagent/编排设计实践：**  
- 只在"需要上下文隔离"或"需要并行"时才引入 subagent，否则就是纯粹的调度开销 —— deterministic 的重复工作留在 skill 层用循环处理即可，不要为每次调用都 spawn 一个新 subagent。  
- Subagent 之间的研究路径如果彼此不依赖（比如同时排查代码、日志、系统资源三条线索），并行 subagent 收益最大；如果有依赖关系（比如必须先拿到诊断结果才能出优化建议），还是老实串行执行 skill。  
- 给 subagent 一个清晰的"只返回结论摘要"的约束，避免污染主上下文。  
  
**判断优先级（如果 skill 数量已经远多于 MCP/subagent 数量，说明方向是对的）** ：把知识沉淀为 skill 的性价比通常高于为同一件事反复 spawn subagent —— 只有当你确实需要独立上下文或并行时，subagent 才值得付出那份开销。

## 思维变了, 下面就是设计流程和交付标准
  
从"优秀员工"切到"管理者"，**不会自动让你轻松下来**。管理者的累是另一种累 —— 从"亲自执行"变成"设计流程 + 审核结果"。如果编排层设计得不好，你反而会更累：既要盯着每个 subagent 有没有跑偏，又要花心思整合结果，相当于自己又当了一遍总编辑。  
  
真正能让你轻松下来的，不是"用了 agent"这个动作本身，而是下面这几件事有没有做到位：  
  
**1. 你现在累的根源，大概率是"每次都要自己决定用哪个技能、怎么组合"**  
  
比如遇到一个数据库性能问题，你脑子里要过一遍：先跑 `pg-runtime-risk`，再看要不要 `pg-bloat-root-cause`，要不要顺手 `pg-parameter-tuning-advisor`……这个"路由决策"本身就是脑力消耗，而且是**每次重新做一遍**，没有被沉淀下来。  
  
这部分工作恰恰是可以让 Claude 自己干的 —— 写一个调度层（可以是一个 skill，也可以是一个 subagent），输入是"用户症状描述"，输出是"该调用哪几个专项技能、以什么顺序"。这一步做完，你从"决策者"变成了"提需求的人"，这才是真正的减负，而且它不一定需要 subagent，一个好的调度 skill 就够。  
  
**2. 真正该交给 subagent 的，是"你现在需要盯着看 但 其实不需要你盯着"的部分**  
  
比如 `pg-load-spike-forensics` SKILL 那种九步根因分析，你现在如果是自己在主会话里一步步跟着看每条 SQL 结果，那这部分注意力消耗是可以转移的 —— 让它在独立 subagent 里跑完九步，只把"根因是什么、证据链是什么"这个结论抛给你。你不需要看它怎么一步步排查的，只需要看结论对不对。 **省的是"过程陪跑"的精力，不是决策的精力** —— 决策该不该采纳这个结论，还得是你。  
  
**3. 管理者真正轻松的前提是"信任 + 可验证的输出契约"**  
  
如果你对 subagent 返回的结果每次都要从头核实一遍（因为不放心），那你比自己干还累。所以这里的关键投入是：给每个 subagent/skill 定义清楚的执行逻辑、输出格式和自检项（比如你的报告要求"结论 + 证据链 + 置信度"），这样你审核的时候扫一眼结构化的东西就能判断靠不靠谱，而不是重新推导一遍逻辑。这个"契约设计"是一次性投入，换来的是长期的省心。  
  
**比如我最喜欢的框架是这样的: 结论/观点、逻辑推演过程、支撑结论/观点的证据链、证据权威性、结论适合的边界、结论成立的前提条件(使用第一性原理拆解)、前提发生变化时的新结论、结论的置信度等.**  
  
**所以: 我该把'决策路由' 和 '过程陪跑'这两件事外包出去，自己只保留'设计流程', '设计可验证的规范的契约', '定义问题'和'审核结论'这几个动作。**  
  
这个转变，subagent 是工具，但真正省力的是你有没有花时间把"调度逻辑"和"输出契约"写清楚、写死。如果只是把原来自己顺序跑的技能改成并行丢给几个 subagent，而你还是每个都要仔细看一遍 —— 那顶多算是把体力活换成了监工活，未必真的轻松。

### 一、三层架构，缺一不可  
  
```  
用户症状描述  
    ↓  
【路由层】一个调度 skill/subagent，判断该调用谁、什么顺序  
    ↓  
【执行层】专项 skill（可选 subagent 隔离执行）  
    ↓  
【契约层】统一输出格式，供路由层整合、供你审核  
    ↓  
汇总结论 → 你  
```  
  
你现在缺的不是执行层（你的 `pg-*` 已经很扎实），缺的是路由层和契约层 —— 这两层没有，所以你自己在当"路由器"和"格式转换器"，这才是累的真正来源。  
  
### 二、路由层：把"该调用谁"这个决策也变成一份可复用资产  
  
具体做法是写一个入口技能，比如叫 `pg-triage`，它不做任何诊断本身，只做**症状 → 技能组合**的映射决策。可以设计成一张决策表放在 SKILL.md 里：  
  
| 用户症状关键词 | 应调用的专项技能 | 执行关系 |  
|---|---|---|  
| 突然变慢/load spike | `pg-load-spike-forensics` | 独立，建议 subagent |  
| 磁盘暴涨/表变大 | `pg-find-bloat` + `pg-large-table-optimize` | 串行（先诊断再优化建议） |  
| 上线前检查 | `pg-sql-audit` + `pg-design-audit` + `pg-security-audit` | 并行，互不依赖 |  
| 例行体检 | `pg-runtime-risk` + `pg-host-resource-risk` + `pg-parameter-tuning-advisor` | 并行 |  
| 具体慢 SQL | `pg-sql-tuning-advisor` + `pg-top-sql-analyze` | 串行 |  
  
这张表就是你脑子里那份"决策疲劳"的外化。写一次，之后每次遇到同类症状，Claude 自己查表决定路由，你不用再想"这次该跑哪几个"。  
  
**关键点**：路由层的输出不是诊断结论，而是"执行计划" —— 即"调用 A、B 并行，C 依赖 B 的结果"。这一步做完，你已经从"决策者"退到"审批执行计划的人"。  
  
### 三、哪些执行层该 subagent 化：一个简单判断标准  
  
不是所有 `pg-*` 都需要变成 subagent，判断标准就三条，任意一条满足就该隔离：  
  
1. **中间产物很多、你不需要全程盯着** —— 比如 `pg-load-spike-forensics` 九步分析里的每一条 SQL 查询结果，你只关心最后的根因链条，不关心第三步查了什么视图。这种适合 subagent，只把摘要吐回来。  
2. **可以和别的诊断并行、互不依赖** —— 比如上线前的 `pg-sql-audit`、`pg-design-audit`、`pg-security-audit` 三者互不依赖，完全可以三个 subagent 同时跑，比你自己串行等三次要快，也不用你分神切换上下文。  
3. **执行时间长，你会中途想去干别的事** —— 长耗时的扫描类任务（`pg-find-unused-index` 跨全库扫描），丢给 subagent 后台跑，你先处理别的，等结果通知你。  
  
反过来，像 `pg-parameter-tuning-advisor` 这种需要你和 Claude 反复讨论权衡（比如"这台机器内存有限，到底该不该调大 shared_buffers"）的，**不适合 subagent** —— 因为这类任务的价值恰恰在于交互式讨论，隔离上下文反而增加沟通成本。  
  
### 四、契约层：统一输出格式，这是真正省审核精力的地方  
  
给所有专项技能（不管是不是 subagent 执行）定义同一套返回结构，比如：  
  
```markdown  
## 结论 / 观点  
[一句话：症状是什么、根因判断是什么]  
  
## 逻辑推演过程  
[从现象到结论的推导链条，非结论、非证据本身，而是"证据如何一步步导出结论"的推理路径。  
例：checkpoint 频繁 → 触发大量 full page write → IO 争抢 → 与卡顿时间窗口吻合 → 排除锁等待/网络因素后，收敛到 checkpoint 参数配置不当]  
  
## 证据链  
1. [具体数据/查询结果] → 支持了什么判断  
2. [具体数据/查询结果] → 支持了什么判断  
...  
（每条证据须可回溯到具体 SQL/日志/快照来源，不接受"经验判断"类无源证据）  
  
## 证据权威性  
| 证据 | 来源类型 | 权威性 |  
|---|---|---|  
| 证据1 | 实时快照直接查询 | 高（一手数据） |  
| 证据2 | 用户口述症状描述 | 中（未验证的间接信息） |  
| 证据3 | 历史经验类比 | 低（非本次实测，仅供参考） |  
（区分"一手实测数据" vs "推断/类比" vs "未经验证的用户描述"，避免把强弱不同的证据混为一谈）  
  
## 结论适合的边界  
[这个结论在什么范围内成立：适用的 PG 版本、部署架构（单机/HA/云托管）、负载特征等。  
明确说明"本结论不适用于"的场景，防止误用到不匹配的环境]  
  
## 结论成立的前提条件（第一性原理拆解）  
[把结论拆解到最基础的假设层，列出缺一不可的前提。  
例：  
- 前提1：checkpoint_timeout 与 max_wal_size 当前配置为默认值（未被自定义调整）  
- 前提2：磁盘 IO 是本次瓶颈的唯一变量（假设网络、CPU 均正常）  
- 前提3：卡顿期间无并发大事务/批量写入]  
  
## 前提发生变化时的新结论  
[针对上面每条前提，给出"如果前提不成立，结论会如何变化"，形成条件分支而非单一死结论。  
例：  
- 若前提2不成立（发现同期有批量导入任务）→ 根因应指向应用层批处理调度，而非参数配置，建议动作随之改为"调整批处理时间窗口"而非"调整 checkpoint 参数"  
- 若前提3不成立（发现长事务持锁）→ 应转向 `pg-find-bloat` / 锁等待分析，当前结论仅为并发因素之一，非唯一根因]  
  
## 置信度  
高/中/低，理由：[基于证据权威性和前提验证完整度综合判断 —— 一手实测证据越多、前提验证越充分，置信度越高；仅靠间接推断或前提未逐一验证，置信度应主动降级]  
  
## 建议动作  
- [ ] 立即执行：xxx（低风险，前提条件已充分验证）  
- [ ] 需要你确认后执行：xxx（有数据丢失/锁表风险，或前提条件尚未完全排除其他可能）  
- [ ] 仅供参考：xxx（置信度中低，或依赖未验证前提）  
  
## 未覆盖 / 需人工判断的点  
[技能力所不及、需要你亲自决策的部分 —— 尤其是那些无法用只读查询验证的前提假设，或需要业务上下文才能判断的边界情况]  
```  
  
这套契约的价值在于：你审核一份报告的时候，不需要重新推导逻辑，只需要扫"证据链站不站得住"和"未覆盖的点"这两处。10 份报告审核下来，你花的是"复核"的精力，不是"重新分析"的精力 —— 这是质变。  
  
你现有的 `pg-*` 已经有"Chinese output + 分级报告"的共性，只差把这套契约固化成一个**共享的输出规范文件**（比如 `pg-output-contract.md`），所有 `pg-*` 技能统一 include/引用它，而不是每个技能各写各的报告格式。这是一次性投入，长期省心。  
  
### 五、走一遍完整例子  
  
假设你收到反馈："客户数据库昨晚突然卡顿，现在恢复了，但不知道为什么"。  
  
- **你要做的**：把这句话丢给 `pg-triage`  
- **路由层判断**：这是"突然变慢+已恢复"模式 → 调用 `pg-load-spike-forensics`（subagent，因为过程繁琐、已经结束不需要实时盯）+ 并行调用 `pg-runtime-risk` 兜底扫描当前状态  
- **执行层**：两个 subagent 各自跑完，各自按契约格式返回  
- **路由层整合**：把两份报告的"结论"部分拼一起，交叉验证证据链是否一致，生成一份总报告  
- **你看到的**：一份总报告，结论 + 证据链 + 需要你确认的动作项（比如"建议调大 checkpoint 相关参数，需你确认目标机器内存余量"）  
  
你全程只做了两件事： **提需求**（丢一句话）、**审结论**（看那个需要确认的参数改动）。中间九步排查、两边扫描的过程,你完全没有参与。  
  
做完这个思维转变和 Claude 工具的使用转变后, 你的决策时间应该趋近于零（因为路由表已经写好了），你花的时间应该是纯粹在"审核这次的结论有没有异常"上。如果发现自己还是在每次重新想"这次该跑哪个技能组合"，说明路由层没沉淀下来，還是员工思维在跑；如果发现自己在核对 subagent 报告时又把原始数据全翻了一遍，说明契约层没起作用，你还是自己在当校对员。  
  
这两个信号是你判断"管理者转型有没有成功"的抓手，比"我用没用 subagent"这个表面指标准得多。
