## 背景  
  
  
  
# PostgreSQL 有诸多 Stable branch, 如何找到每个 Stable branch 与 master branch 的分叉点?  
  
PostgreSQL 官方仓库里 `REL_16_STABLE`、`REL_15_STABLE` 等各个 stable 分支各自是从 master 的哪个 commit 切出去的。这个可以直接用 `git merge-base` 批量算出来，非常干净。  
  
  
## 核心命令  
  
```bash  
git clone https://github.com/postgres/postgres.git  
cd postgres  
  
# 拉全部远程分支  
git fetch origin  
  
# 列出所有 stable 分支  
git branch -r | grep STABLE  
```  
  
## 批量找每个 stable 分支的分叉点  
  
```bash  
for br in $(git branch -r | grep STABLE | sed 's/origin\///'); do  
  fork_point=$(git merge-base origin/master origin/$br)  
  echo "=== $br ==="  
  git show --no-patch --format="  分叉 commit: %H%n  日期: %ci%n  说明: %s" $fork_point  
  echo  
done  
```  
  
输出大概长这样(示意):  
  
```  
=== REL_17_STABLE ===  
  分叉 commit: 7dcc6f8e6d7a0eb0ce90802311278723843b4bbd  
  日期: 2024-07-01 07:35:01 +0900  
  说明: Run pgperltidy  
  
=== REL_18_STABLE ===  
  分叉 commit: 9c5b9a280cb6089c011a01797868da83f97d0230  
  日期: 2025-06-29 21:43:39 -0400  
  说明: Do pre-release housekeeping on catalog data.  
  
=== REL_19_STABLE ===  
  分叉 commit: 9cfd19bc10ac07139ca6c6d051d4492764441edb  
  日期: 2026-06-29 15:33:52 -0400  
  说明: Add previous 2 commits to .git-blame-ignore-revs.  
```  
  
## 需要注意的几个坑  
  
**1. `git merge-base` 找的是"当前时刻"两条线的最近公共祖先，不一定是"branch cut"当天的那个 commit**  
  
因为 master 后续可能会有 cherry-pick 回 stable 分支的 commit（PostgreSQL 官方确实经常把 bugfix 从 master cherry-pick 到多个 stable 分支），这些 cherry-pick 提交在 stable 分支上会生成新的 commit hash（内容相同但 hash 不同，因为 parent 不同），所以理论上不影响 merge-base 结果——只要 stable 分支没有反向 merge 回 master（PostgreSQL 官方历史确实不会这样做），`merge-base` 依然准确。  
  
所以对 PostgreSQL 这种"只从 master 单向切分支、之后不合并回去"的规范仓库，`merge-base` 是可靠的。  
  
**2. 一次性生成全部分支的分叉点时间线（更直观）**  
  
```bash  
for br in $(git branch -r | grep STABLE | sed 's/origin\///' | sort -t_ -k2 -n); do  
  fork_point=$(git merge-base origin/master origin/$br)  
  date=$(git show -s --format=%ci $fork_point)  
  echo "$br  ->  $date  ($fork_point)"  
done  
```  
  
这样可以按分支号排序，直接看到每个大版本是哪天从 master 切出去的，也能反推出 PostgreSQL 每年大版本的开发周期节奏（一般每年 5 月左右 feature freeze 前后切分支，9-10 月正式发布）。  
  
输出大概长这样(示意):  
  
```  
REL_17_STABLE  ->  2024-07-01 07:35:01 +0900  (7dcc6f8e6d7a0eb0ce90802311278723843b4bbd)  
REL_18_STABLE  ->  2025-06-29 21:43:39 -0400  (9c5b9a280cb6089c011a01797868da83f97d0230)  
REL_19_STABLE  ->  2026-06-29 15:33:52 -0400  (9cfd19bc10ac07139ca6c6d051d4492764441edb)  
```  
  
# 如何只克隆某些选定的 branch?  
  
可以用 `git clone` 的分支过滤参数，只拉取指定分支的历史，不用 clone 全部分支。  
  
## 方法一：clone 时用 `--branch` + `--single-branch`（但只能拉一个分支）  
  
`git clone` 本身一次只能指定一个 `--branch`，多分支需要分两步：先浅 clone 一个分支，再补拉其他分支。  
  
```bash  
# 先 clone master（浅克隆，不带全部分支）  
git clone --single-branch --branch master https://github.com/postgres/postgres.git  
cd postgres  
  
# 再依次追加其他分支  
git remote set-branches --add origin REL_19_STABLE  
git remote set-branches --add origin REL_18_STABLE  
git remote set-branches --add origin REL_17_STABLE  
  
git fetch origin  
```  
  
这样 fetch 下来后本地会有 `origin/master`、`origin/REL_19_STABLE`、`origin/REL_18_STABLE`、`origin/REL_17_STABLE` 四个远程分支，且**不会**拉取其他几十个历史 stable 分支（如 REL_9_6_STABLE 等），显著节省时间和空间。  
  
## 方法二：一步到位（推荐，更简洁）  
  
`git clone` 支持 `--no-single-branch` 配合初始不指定分支，然后用 `remote.origin.fetch` 精确控制要跟踪的 refspec：  
  
```bash  
mkdir postgres && cd postgres  
git init  
git remote add origin https://github.com/postgres/postgres.git  
  
# 精确设置只跟踪这几个分支  
git config remote.origin.fetch "+refs/heads/master:refs/remotes/origin/master"  
git config --add remote.origin.fetch "+refs/heads/REL_19_STABLE:refs/remotes/origin/REL_19_STABLE"  
git config --add remote.origin.fetch "+refs/heads/REL_18_STABLE:refs/remotes/origin/REL_18_STABLE"  
git config --add remote.origin.fetch "+refs/heads/REL_17_STABLE:refs/remotes/origin/REL_17_STABLE"  
  
git fetch origin  
git checkout master  
```  
  
这个方法的好处是配置持久化在 `.git/config` 里，以后 `git fetch` 会自动只更新这 4 个分支，不会意外拉到别的分支。  
  
## 如果还想加上浅历史（--depth）省流量  
  
如果你只是要做 `merge-base` 分叉点分析，理论上需要**完整历史**才能保证 merge-base 计算准确（浅克隆会导致 merge-base 找不到真正的公共祖先，可能报错或给出错误结果）。  
  
所以这里**不建议加 `--depth`** ，否则后续 `git merge-base origin/master origin/REL_17_STABLE` 很可能因为历史被截断而找不到正确的分叉点，甚至直接报错 "fatal: no merge base found"。  
  
如果你只是想看代码本身、不需要做分叉点分析，可以用 `--depth=1` 进一步加速：  
  
```bash  
git clone --single-branch --branch master --depth=1 https://github.com/postgres/postgres.git  
```  
  
## 验证拉到的分支 & 后续操作  
  
```bash  
git branch -r  
  
# 应该只看到:  
# origin/master  
# origin/REL_19_STABLE  
# origin/REL_18_STABLE  
# origin/REL_17_STABLE  
  
# 然后就可以直接用之前的分叉点脚本  
for br in $(git branch -r | grep STABLE | sed 's/origin\///'); do  
  fork_point=$(git merge-base origin/master origin/$br)  
  echo "=== $br ==="  
  git show --no-patch --format="  分叉 commit: %H%n  日期: %ci%n  说明: %s" $fork_point  
  echo  
done  
```
