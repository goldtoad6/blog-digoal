## 背景 
概述

PostgreSQL 通过实现 **SQL/PGQ**（Property Graph Queries，属性图查询）支持图数据功能，这是 ISO/IEC 9075-16 标准的一部分。属性图允许在现有关系表之上定义图结构，并使用图模式匹配进行查询，而无需改变底层数据存储方式。

**核心特性：**
- 数据保留在关系表中，但以图的形式暴露
- 关系查询和图查询使用相同的执行基础设施
- 可以在单一查询中混合使用关系型和图查询

 

## 核心概念

| 概念 | 描述 |
|------|------|
| **顶点（Vertex）** | 表示实体表（节点） |
| **边（Edge）** | 表示关系表（顶点之间的有向连接） |
| **标签（Label）** | 定义顶点或边的"类型"（类似于图术语中的表名） |
| **属性（Property）** | 与顶点或边关联的数据（从列暴露） |

### 记忆技巧
- 顶点：用圆括号 `()` 表示
- 边：用方括号 `[]` 表示
- 方向箭头：`->` 或 `<-`

 

## 创建属性图

### 基本语法

```sql
CREATE PROPERTY GRAPH graph_name
    VERTEX TABLES (
        table_name [LABEL label_name] [KEY (column)] [PROPERTIES (column [AS alias], ...)]
    )
    EDGE TABLES (
        table_name 
            [KEY (key_column)]
            SOURCE KEY (source_col) REFERENCES source_table (ref_col)
            DESTINATION KEY (dest_col) REFERENCES dest_table (ref_col)
            [LABEL label_name]
    );
```

### 组件说明

**顶点表（VERTEX TABLES）：**
- `LABEL`：指定图的顶点标签名（默认为表名）
- `KEY`：指定顶点表的键列
- `PROPERTIES`：指定暴露为属性的列

**边表（EDGE TABLES）：**
- `SOURCE KEY ... REFERENCES`：指定源顶点表及其键列
- `DESTINATION KEY ... REFERENCES`：指定目标顶点表及其键列
- `LABEL`：指定边的标签名

 

## 完整示例

### 1. 创建基础表

```sql
CREATE TABLE products (
    product_no integer PRIMARY KEY,
    name varchar,
    price numeric
);

CREATE TABLE customers (
    customer_id integer PRIMARY KEY,
    name varchar,
    address varchar
);

CREATE TABLE orders (
    order_id integer PRIMARY KEY,
    ordered_when date
);

CREATE TABLE order_items (
    order_items_id integer PRIMARY KEY,
    order_id integer REFERENCES orders (order_id),
    product_no integer REFERENCES products (product_no),
    quantity integer
);

CREATE TABLE customer_orders (
    customer_orders_id integer PRIMARY KEY,
    customer_id integer REFERENCES customers (customer_id),
    order_id integer REFERENCES orders (order_id)
);
```

### 2. 基本属性图定义

```sql
CREATE PROPERTY GRAPH myshop
    VERTEX TABLES (
        products,
        customers,
        orders
    )
    EDGE TABLES (
        order_items SOURCE orders DESTINATION products,
        customer_orders SOURCE customers DESTINATION orders
    );
```

### 3. 使用自定义标签

```sql
CREATE PROPERTY GRAPH myshop
    VERTEX TABLES (
        products LABEL product,
        customers LABEL customer,
        orders LABEL "order"
    )
    EDGE TABLES (
        order_items SOURCE orders DESTINATION products LABEL contains,
        customer_orders SOURCE customers DESTINATION orders LABEL has_placed
    );
```

### 4. 多标签支持

同一个顶点或边可以有多个标签，便于从不同角度查询数据：

```sql
CREATE TABLE employees (
    employee_id integer PRIMARY KEY,
    employee_name varchar
);

CREATE PROPERTY GRAPH myshop
    VERTEX TABLES (
        products LABEL product,
        customers LABEL customer LABEL person PROPERTIES (name),
        orders LABEL order,
        employees LABEL employee LABEL person PROPERTIES (employee_name AS name)
    )
    EDGE TABLES (
        order_items SOURCE orders DESTINATION products LABEL contains,
        customer_orders SOURCE customers DESTINATION orders LABEL has
    );
```

 

## 图查询：GRAPH_TABLE 与 MATCH

### 基本语法

```sql
SELECT columns
FROM GRAPH_TABLE (
    graph_name
    MATCH (pattern)
    COLUMNS (expression AS alias, ...)
);
```

### GRAPH_TABLE 组件

1. **图名称**：要查询的属性图名称
2. **MATCH 关键字**：包含要匹配的图模式
3. **图模式表达式**：定义顶点、边和它们之间的关系
4. **COLUMNS 关键字**：定义输出列

### MATCH 模式语法

| 模式 | 描述 |
|------|------|
| `()` | 匹配单个顶点 |
| `()->()` | 匹配 顶点 → 边 → 顶点 |
| `( ) - [ ] - > ( )` | 同上（带空格） |

### 标签匹配

使用 `IS` 关键字匹配标签：

```sql
-- 匹配 person 顶点
(IS person)

-- 匹配 person → account 通过 'has' 边
(IS person)-[IS has]->(IS account)

-- 多个边标签（或语义）
(IS person)-[IS has]->(IS account|creditcard)
```

### 方向

边默认是有向的：

```sql
-- 正向
(IS person)-[IS has]->(IS account)

-- 反向
(IS account)<-[IS has]-(IS person)

-- 任意方向
(IS person)-[IS is_friend_of]-(IS person)
```

### 过滤条件

在元素模式内使用 `WHERE` 子句：

```sql
(IS person)-[IS has]->(a IS account WHERE a.type = 'savings')
```

### 模式变量

将变量绑定到匹配的元素以在 `COLUMNS` 子句中使用：

```sql
GRAPH_TABLE (
    mygraph MATCH (p IS person)-[h IS has]->(a IS account)
    COLUMNS (
        p.name AS person_name,
        h.since AS has_account_since,
        a.num AS account_number
    )
)
```

 

## 查询示例

### 查询今日下单的客户

```sql
SELECT customer_name 
FROM GRAPH_TABLE (
    myshop 
    MATCH (c IS customers)-[IS customer_orders]->(o IS orders WHERE o.ordered_when = current_date) 
    COLUMNS (c.name AS customer_name)
);
```

**等价的关系查询：**

```sql
SELECT customers.name 
FROM customers 
JOIN customer_orders USING (customer_id) 
JOIN orders USING (order_id) 
WHERE orders.ordered_when = current_date;
```

### 使用多标签查询

```sql
SELECT * 
FROM GRAPH_TABLE (
    myshop 
    MATCH (IS person WHERE name = '张三')-[*]->(IS order) 
    COLUMNS (person.name AS person_name, order.order_id AS order_number)
);
```

### 查询所有属性

```sql
SELECT * 
FROM GRAPH_TABLE (
    myshop 
    MATCH (src)-[conn]->(dest) 
    COLUMNS (
        src.name AS source_name,
        conn.quantity AS edge_quantity,
        dest.name AS dest_name
    )
);
```

 
## 实现原理

### 执行机制

`GRAPH_TABLE` 构造在查询中扮演表函数的角色，返回基于图模式匹配的行集。其实际执行过程为：

1. **重写阶段**：`GRAPH_TABLE` 表达式由重写器展开为关系连接树
2. **规划阶段**：生成的连接树传递给查询规划器
3. **执行阶段**：使用标准的 PostgreSQL 执行器处理

这意味着图查询和关系查询使用相同的底层优化和执行基础设施，可以无缝混合使用。

### 视图特性

SQL/PGQ 功能在 PostgreSQL 中实现为**只读视图**，实际数据仍存储在表中，只是以图的形式暴露供查询使用。

 

## 关键要求

1. **主键要求**：表必须有主键，或通过 `KEY(column)` 明确指定
2. **外键要求**：边表需要外键约束，或通过 `SOURCE/DESTINATION KEY` 子句详细指定
3. **属性匹配**：当多个表共享标签时，属性必须在数量、名称和类型上匹配

 

## 最佳实践

### 1. 合理设计顶点表

- 选择频繁作为查询起点的实体作为顶点
- 确保顶点表有明确的主键

### 2. 边表设计

- 边表应包含外键引用源顶点和目标顶点
- 使用描述性的边标签名称

### 3. 查询优化

- 利用 `WHERE` 子句在模式匹配阶段过滤数据
- 避免不必要的全图遍历

### 4. 混合查询

```sql
-- 结合关系查询和图查询
SELECT g.customer_name, o.total_amount
FROM GRAPH_TABLE (
    myshop MATCH (c IS customers)-[IS has_placed]->(o IS orders)
    COLUMNS (c.name AS customer_name, o.order_id AS order_id)
) g
JOIN orders o ON o.order_id = g.order_id
WHERE o.total_amount > 1000;
```

 
## 总结

PostgreSQL 的属性图功能为关系数据提供了强大的图查询能力，无需改变现有数据结构。通过 `CREATE PROPERTY GRAPH` 定义图结构，使用 `GRAPH_TABLE` 和 `MATCH` 进行图模式匹配，开发者可以在保持关系型数据库优势的同时，享受图查询的灵活性。

这一特性特别适用于：
- 社交网络分析
- 推荐系统
- 欺诈检测
- 路径优化
- 层次数据结构查询
