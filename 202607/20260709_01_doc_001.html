1.1 · 起点

先分清 Workload

参数调优的起点不是参数本身,是你的 workload 形态。三种 workload,三套打法。

⚡
OLTP
短事务高并发点查ms 级

work_mem 小 · shared_buffers 适中 · random_page_cost ↓

关注: TPS · P99 · 锁等待 · 长事务
📈
OLAP
长查询大表扫并行s-m 级

work_mem 大 · 并行 worker 多 · random_page_cost 高

关注: 查询 RT · 临时文件 · worker 利用率
🔀
HTAP 混合
长短并存资源争抢

资源池隔离 · OLAP 单独实例 · Read Replica

关注: 两套指标同时监控
上来就照搬"标准 postgresql.conf" → OLTP 库 work_mem 给大了浪费内存,OLAP 库 work_mem 给小了疯狂落盘。workload 不分清,所有调优都是猜。

1.3 · 事前优化 · 环境

OS 与硬件的硬功夫

这些坑,参数再调也救不回来。

🔧 SSD 4K 对齐
物理页 4K · 文件系统块 · PG block 8K 都要对齐
否则一次写触发 read-modify-write,性能掉一半

证伪: fio randwrite -bs=4k,未对齐比对齐低 30%~50%

mkfs.xfs -d su=4k,sw=1
⚙️ IO 调度器
SSD 没有磁头,cfq/deadline 反而增加延迟
Linux 5.x 默认 mq-deadline / none / kyber 更适合

边界: 云盘由云厂处理,看不见也改不了

echo none > /sys/block/nvme0n1/queue/scheduler
🧠 HugePage
shared_buffers 推荐 RAM × 1/4
超过 32GB 必须配 HugePage,否则 TLB miss 让大库掉 30%+

echo $((64*1024*1024/2048)) > /proc/sys/vm/nr_hugepages

边界: 容器嵌套虚拟化时配不上
📊 块设备条带化
多盘 RAID0 把单盘 IOPS × N
chunk size 64KB~256KB,与 PG block 8K 对齐

mdadm -C /dev/md0 -l 0 -n 4 /dev/nvme[0-3]n1

边界: 云盘已底层条带化,不需要再做 RAID0
必避坑:禁用 btrfs(社区历史问题多)、zfs(PG 18+ 适配中)。PG 官方推荐 xfs。

内核脏页参数

# 128GB 内存示例:让刷盘更平滑
vm.swappiness = 0                    # PG 用 swap = 性能灾难
vm.dirty_background_bytes = 314572800  # 300MB 触发后台刷
vm.dirty_bytes = 4294967296          # 4GB 强制同步
vm.dirty_expire_centisecs = 500      # 5 秒(默认 30s 太长)
vm.dirty_writeback_centisecs = 100   # 1 秒唤醒
net.core.somaxconn = 4096             # 高并发排队
1.3 · 补充 · 调度器对比

IO 调度器 · 6 种核心区别

调度器是块设备层的请求重排器。机械盘时代为磁头寻道而生,SSD/NVMe 时代多数调度器反而成了开销。

↑ 按盘类型选调度器 · 一棵树搞定

6 种调度器 · 关键差异

调度器队列核心策略适用PG 场景评价
noop 单队列 几乎不排序,只合并相邻请求 NVMe · 虚拟化透传 ✅ 延迟最低
deadline 单队列 读/写各设 500ms 截止,防饥饿 SATA SSD · 混合负载 ✅ 兜底之选
cfq 单队列 按进程公平时间片轮转 桌面(已废弃) ❌ Linux 5.x 已移除
mq-deadline 多队列 blk-mq deadline 的并行版,每个 CPU 一队列 SATA SSD · Linux 5.x 默认 ✅ 通用首选
kyber 多队列 blk-mq 读/写分队列 + 目标延迟控制 NVMe · 快速设备 ✅ NVMe 优选
bfq 多队列 blk-mq 按进程 IO 预算公平,适合桌面 桌面交互 ❌ 服务器不推荐
关键差异一句话总结:
  • noop = 啥也不干,合并相邻请求后直接送,适合"设备自己就够快"
  • deadline = 防饿死型,保证请求有最迟完成时间
  • cfq = 公平型,进程轮流用 IO,2019 起被 Linux 内核废弃
  • mq-deadline = deadline 的多核并行版,Linux 5.x 默认
  • kyber = 读/写分家 + 目标延迟,快设备的精细调度
  • bfq = 给进程的"IO 预算",桌面才需要,服务器会拖累吞吐

切换命令 · 一行搞定

# 查看当前可用调度器(方括号 [ ] 包裹的就是当前)
$ cat /sys/block/nvme0n1/queue/scheduler
[none] mq-deadline kyber bfq

# 临时切换(重启失效)
echo none > /sys/block/nvme0n1/queue/scheduler

# 永久:写入 udev 规则(开机自动应用)
$ cat /etc/udev/rules.d/60-ssd-scheduler.rules
ACTION=="add|change", KERNEL=="nvme[0-9]n[0-9]", ATTR{queue/scheduler}="none"
ACTION=="add|change", KERNEL=="sd[a-z]", ATTR{queue/scheduler}="mq-deadline"

$ udevadm control --reload-rules   # 重载规则
边界 / 证伪:
  • 云盘看不见:EBS / ESSD / 阿里云 SSD 都是云厂在宿主机里调好,你看不到也改不了 — 用 iostat -x 1 观察 r_await/w_await 验证
  • 改完有用没用:跑 fio -filename=/pgdata/test -direct=1 -rw=randwrite -bs=4k -size=1G -runtime=30 对比 IOPS / 延迟
  • 不能跨设备套用:NVMe 用了 bfq 会从 50 万 IOPS 掉到 5 万,一定要按盘类型选
📋 调度器速记
  • NVMe → none 或 kyber · 设备自己够快,别加开销
  • SATA SSD → mq-deadline · Linux 5.x 默认,平衡之选
  • HDD 机械盘 → mq-deadline · 合并寻道 + 防饿死
  • 云盘 → 不可控,看 iostat 延迟就行
  • 永远不要在 NVMe 上用 bfq / cfq
1.4 · 事前优化 · 参数

必杀参数速记

每条公式都是经验值,不是教条。上生产前在自己环境跑 benchmark 验证。

🧮 连接与内存
max_connections = 2000
公式: 内存(GB) × 1000 × 1/4 ÷ 5

shared_buffers = RAM × 1/4(配 HugePage)
work_mem = min(物理内存/4096, 64MB)
单条 query 多个 NODE 都用,实际 ×N 倍消耗

effective_cache_size = RAM × 1/2
给 OS page cache 的估算
📀 WAL 与复制
wal_compression = lz4
max_wal_size = 16GB
checkpoint_timeout = 30min
checkpoint_completion_target = 0.9
PG 14+ 默认,拉长 checkpoint 把 IO 摊平

synchronous_commit = on
off 模式最坏丢 3 × wal_writer_delay
🧹 Autovacuum 关键
autovacuum_max_workers = 8
默认 3,大库不够。公式:max(min(8, CPU/2), 5)

autovacuum_vacuum_scale_factor = 0.05
默认 0.2 → 大表等到 20% 才触发,膨胀元凶

autovacuum_vacuum_cost_delay = 10ms
autovacuum_freeze_max_age = 200000000
不要改!
🎯 规划器
random_page_cost = 1.1 本地 SSD
random_page_cost = 1.0 NVMe 本地
random_page_cost = 1.5 云盘
默认 4.0 是机械盘时代定的

effective_io_concurrency = 200
default_statistics_target = 500
jit = on(OLAP 受益,OLTP 关掉省 CPU)
证伪: EXPLAIN 看是否多走 Index Scan。改 random_page_cost=1.1 后某些大表 Seq Scan 反而快 —— 这是 Planner"看穿"了索引的真正价值,不是失准。
1.5 · 事前优化 · 连接池

连接 = 进程,雪崩起点

PG 单进程模型下,5000 连接就是 5000 个进程。生产单实例后端连接数控制在 100~500。

↑ PgBouncer transaction 模式复用连接,把"100 个客户端 / 1 个进程"的浪费压扁
session 模式
一 client 占一 server 直到断开
完全不 pool,没意义
transaction 模式 ✓
一 client 占 server 直到事务结束
通用首选
不能:session SET / prepared / LISTEN / temp table
statement 模式
一条 SQL 一个 server
事务边界不保,别用
1.6 · 事前优化 · 业务

业务优化 = 性价比最高

上面所有环境/参数调优加起来,可能都比不上业务层改一个 SQL。

类型精准 · 一表打尽

索引反模式 · 8 条清单

🚫 每个查询都建索引
写性能下降 / 维护成本高
🚫 复合索引列顺序错
(a, b) 与 (b, a) 完全不同
🚫 长字段建索引
VARCHAR(255) 让 B-Tree 高度 +1
🚫 冗余索引
(a) 和 (a, b) 同时存在
🚫 死索引不清理
idx_scan=0 持续 1 个月,DROP CONCURRENTLY
✅ Partial Index
1% 数据占 99% 查询 · 投入产出比最高

EXPLAIN · 6 信号法

1. actual time
真耗时 · loops 相乘才是真实成本
2. rows vs actual
偏差 > 10× → 统计信息失真
3. Buffers hit/read
read 占比高 → shared_buffers 不够
4. loops > 1
多半是 NestLoop,成本完全骗人
5. 节点类型
Seq Scan / Index / Bitmap / Hash / Sort
6. SETTINGS
打印实际生效的优化器参数
必避坑: EXPLAIN ANALYZE 真的执行 SQL!对 INSERT/UPDATE/DELETE 必须包事务: 
BEGIN;
EXPLAIN ANALYZE DELETE FROM t WHERE ...;
ROLLBACK;
1.7 · 事后优化 · 擒贼擒王

TOP SQL = 80% 起点

报警响了?先问一句:是 PG 的锅吗?

擒贼先擒王适用于已经定位到 PG 内、SQL 占比 > 50% 的场景。如果报警来自连接池打满 / JVM Full GC / 上游阻塞,TOP SQL 会因为请求堆积而总耗时虚高 —— 盯着 SQL 改没用,根因在 APP 层。

擒贼擒王 · 一条 SQL

SELECT round(total_exec_time::numeric, 2) total_ms,
       calls,
       round(mean_exec_time::numeric, 2) mean_ms,
       shared_blks_hit + shared_blks_read AS blks,
       query
  FROM pg_stat_statements
 ORDER BY total_exec_time DESC LIMIT 10;

5 大 Planner 失准场景

📊 数据倾斜
rows 估算严重偏高
对策: CREATE STATISTICS ON a,b FROM t
📦 bulk load 后没 ANALYZE
全 Seq Scan
对策: COPY WITH (FREEZE) + 立即 ANALYZE
⏱️ 长时间小批量写入
统计陈旧
对策: autovacuum_analyze_scale_factor = 0.025
📉 数据分布漂移
老统计不反映新常态
对策: 周期性 ANALYZE 或 cron
抓 TOP SQL 后:看 EXPLAIN 七步法(1.6 节)→ 决定改法:
rows 偏差 > 10×→ANALYZE Seq Scan 大表→加索引 Hash Batches > 1→SET work_mem
1.8 · 事后优化 · 锁

锁等待 & 秒杀优化

从一条救命 SQL 说起,到 80 倍性能差距。

🔗 谁堵了谁

SELECT pid, pg_blocking_pids(pid) blocked_by,
       wait_event_type, wait_event, query
  FROM pg_stat_activity
 WHERE wait_event_type = 'Lock';

⚡ SKIP LOCKED · 队列 worker

SELECT * FROM task_queue
 WHERE status = 'pending'
 ORDER BY id LIMIT 1
 FOR UPDATE SKIP LOCKED;

秒杀场景 · 80× 性能差

-- 关键写法:WHERE 里直接抢锁
UPDATE t SET x = x + 1
 WHERE id = ? AND pg_try_advisory_xact_lock(id)
RETURNING *;
前置条件:必须配合"先查再写"的两阶段模式,锁粒度 = 单条记录 id。
边界:若 pg_stat_activity 看到大量 LockWait,说明冲突超出 AD Lock 缓解能力,必须分片 / 排队 / 异步化。
1.9 · 事后优化 · 等待事件

wait_event = 最强诊断武器

告诉你"这一刻 backend 为什么不在跑"。PG 9.6+ 起,9 大类全覆盖。

IO 类

IO:DataFileRead → iostat -x 1, r_await>10ms 物理 IO 瓶颈
IO:WALWrite → 主备延迟?同步复制降级异步
IO:ControlFileSync → checkpoint 风暴?completion_target=0.9

LWLock 类

LWLock:buffer_mapping → shared_buffers 设太大(> RAM 25%)
LWLock:lock_manager → 单实例连接数 > 500,上 pgbouncer
PG 16+ 8 分区优化,buffer_mapping 缓解

Lock 类

Lock:transactionid → 看 pg_blocking_pids,应用层事务顺序不一致
Lock:tuple → 用 SKIP LOCKED 改写
Lock:relation → DDL 阻塞,加 lock_timeout

Activity

state=active 且 wait_event IS NULL → 在 CPU 上跑
大排序 / 哈希聚合
perf top -p <pid> 看是不是 CPU bound

2.1 · 分层架构

先分层,再下钻

4 层漏斗,SQL 层优先 + 漏斗排查补充。

核心观点:SQL 层优先 + 漏斗排查补充。
"数据库慢 90% 不在 SQL 层" —— 听着很爽,但事实是 SQL 层问题确实占大多数生产事故。别被这句话忽悠得漏掉 TOP SQL。
2.2-2.3 · 必开 + 擒贼擒王

5 个开关 + pg_stat_statements

任何一个没开,后续视图就是空的。

⚡ 必开 5 个统计开关

track_activities = on
track_counts     = on
track_io_timing  = on
track_functions  = 'all'
compute_query_id = on  # PG 13+
SELECT pg_reload_conf(); 生效

📝 半必开 3 个日志开关

log_lock_waits = on
log_temp_files = 0     # 0=全部记录
log_min_duration_statement = 3s
OLTP 1~5s · OLAP 30s~5min

pg_stat_statements · 擒贼擒王 SQL

SELECT round(total_exec_time::numeric, 2) total_ms,
       calls, round(mean_exec_time::numeric, 2) mean_ms,
       shared_blks_hit + shared_blks_read AS blks, query
  FROM pg_stat_statements
 ORDER BY total_exec_time DESC LIMIT 10;
关于 queryid:在同一 major 版本内 + 同一 SQL 文本稳定;PG 大版本升级(14→15)会因算法微调而变化 —— 不能说"跨实例"稳定。

外部快照 · 唯一做趋势的方法

pgss 视图 → 定时快照 → 历史表 → 趋势 SQL
单纯靠 pg_stat_statements 没有外部快照就没有趋势 —— 这是新手最常踩的坑。
2.5 · 自动捕获慢 SQL

auto_explain · 慢 SQL 自动落盘

超过阈值的 SQL 自动写执行计划,不用手抓。

shared_preload_libraries = 'auto_explain,pg_stat_statements'
auto_explain.log_min_duration = '3s'
auto_explain.log_analyze      = on
auto_explain.log_buffers      = on
auto_explain.log_format       = 'json'    # PG 10+
auto_explain.sample_rate      = 1
大流量场景(> 5000 QPS)必须抽样 —— sample_rate=0.01~0.1。
log_min_duration=3s + sample_rate=0.01 的组合有盲区:平时 0.5s、突发 4s 的致命 SQL 可能一条都采不到。
生产建议 log_min_duration=1s + sample_rate=0.05 起跳,配合 pg_stat_statements 短期快照双轨。

10 条 救命级 SQL

收藏这 10 条,先救命再优化。

1. 运行中慢 SQL

SELECT pid, now() - query_start dur, state, wait_event, query
  FROM pg_stat_activity
 WHERE state='active' AND now() - query_start > '5s';

2. 长事务

SELECT pid, now() - xact_start xact_age, query
  FROM pg_stat_activity
 WHERE state IN ('active','idle in transaction')
   AND now() - xact_start > '30min';

3. 长空闲事务(真凶)

SELECT pid, now() - state_change idle_age
  FROM pg_stat_activity
 WHERE state LIKE 'idle in transaction%'
   AND now() - state_change > '10min';

4. 长 2PC

SELECT gid, prepared, owner, database
  FROM pg_prepared_xacts
 WHERE now() - prepared > '1h';

5. 谁堵了谁

SELECT pid, pg_blocking_pids(pid) blocked_by,
       wait_event_type, wait_event, query
  FROM pg_stat_activity
 WHERE wait_event_type = 'Lock';

6. 等待事件分布

SELECT wait_event_type, wait_event, count(*)
  FROM pg_stat_activity
 WHERE wait_event IS NOT NULL
 GROUP BY 1,2 ORDER BY count(*) DESC;

7. vacuum 进度

SELECT pid, relid::regclass, phase,
       heap_blks_total, heap_blks_scanned
  FROM pg_stat_progress_vacuum;

8. checkpoint 频率

SELECT * FROM pg_stat_bgwriter;
-- checkpoints_req 远多于
-- checkpoints_timed → 雪崩

9. 复制延迟

SELECT pid, state, sync_state,
       write_lag, flush_lag, replay_lag
  FROM pg_stat_replication;

10. DB 总览 + xid

SELECT datname, numbackends conns,
       round(100.0*blks_hit/
         NULLIF(blks_hit+blks_read,0),2) cache_hit_pct,
       age(datfrozenxid) xid_age
  FROM pg_stat_database;

止血三件套

pg_cancel_backend

只杀当前 query,更安全

SET LOCAL lock_timeout

3.6 · 突发 IO/CPU

后台进程三件套:autovacuum + checkpoint + bgwriter

TOP SQL 没什么变化,但 CPU/IO 高?答案几乎都在这三个里。

🌀 autovacuum 风暴

autovacuum_freeze_max_age = 200000000 是 PG 硬编码防线
某库 age(xid) > 2亿 → 系统强制全表 freeze
"突然很卡"的最大单一原因

🌀 checkpoint 风暴

pg_stat_bgwriter.checkpoints_req ≫ timed
拉长 checkpoint 把 IO 摊平:
completion_target = 0.9
别因为"SSD 快"调到 0.2,反而更尖刺

🌀 bgwriter 不够

buffers_backend > buffers_clean
backend 写盘太多,bgwriter 不够
调大 bgwriter_lru_maxpages 或 shared_buffers

4.1 · SQL 反直觉

为什么 OFFSET 大就慢

OFFSET 不是"跳过",是"先排序再扔"。

SELECT * FROM t
ORDER BY id LIMIT 20
OFFSET 1000000;

先排 1000020 行,扔掉前 1000000
→
SELECT * FROM t
WHERE id > $last
ORDER BY id LIMIT 20;

索引直接定位 20 行
前置条件:固定排序键 + 应用层记上一页最大 ID。
边界:跨页查询("跳到第 1000 页")仍要用 OFFSET —— Keyset 不支持随机跳页。
4.2 · MVCC

为什么表 / 索引会膨胀

结论:膨胀是 vacuum 跟不上的结果,不是 UPDATE/DELETE 本身。

vacuum 跟不上的 7 条原因

1. autovacuum = off
2. scale_factor = 0.2
默认太大,大表等到 20% 才触发
3. naptime = 60s
默认太慢
4. cost_delay = 20ms
默认太斯文
5. max_workers = 3
默认太少
6. maintenance_work_mem 小
7. 长事务 / 2PC / 慢 slot
挡 vacuum

4.5 · XID

为什么需要 FREEZE

32-bit xid 最多 43 亿,PG 设计为循环使用 —— 回卷后"过去"变"未来",数据消失。

4.6 · 索引

为什么 GIN 有时不快

GIN 更新走"快路径"(pending list),vacuum 时才合并。pending list 越大越慢。

  get_raw_page('idx_xxx', 0));
-- 看 n_pending_pages,越大越慢

🛠️ 修复

-- 触发 pending list 合并
第一性原理:GIN 是为"读多写少"设计的,高频写时必须配套高频 vacuum。
