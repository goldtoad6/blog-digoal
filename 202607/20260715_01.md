## 背景  
  
Supabase 的真正价值，一句话概括： **把 PostgreSQL 包装成了"开箱即用的后端"，用 Postgres 原生能力（而不是自造轮子）解决了 Firebase 想解决的问题。**  
  
**1. 对开发者：省掉后端团队**  
  
这些必须的后端功能“Auth、Storage、Realtime、Edge Functions、自动生成的 REST/GraphQL API（PostgREST）”，全部围绕一个 Postgres 实例搭建。前端工程师不用写后端代码就能建产品，这是它对标 Firebase 的核心卖点。  
  
**2. 对原有 PG 用户的价值**  
  
底层就是标准 PostgreSQL，这意味着：  
- 你原有的 SQL、索引、扩展（PostGIS、pgvector、pg_cron 等）经验全部复用，没有 Firebase 那种"锁死在专有 NoSQL 模型"的问题  
- Row Level Security (RLS) 直接用 Postgres 原生权限体系做多租户隔离，而不是应用层加一层网关鉴权  
- 迁移出去的成本低 —— 不像 Firebase 那样数据格式锁死  
  
**3. 商业模式上的价值**  
  
Supabase 本质是在做"开源 + 托管服务"（PostgreSQL 生态的 Vercel）。它的护城河不是技术壁垒（PostgREST、GoTrue 这些组件都开源），而是**把碎片化的 Postgres 生态工具整合打包 + 良好的开发者体验（DX）** ，降低了从 0 到 1 的决策成本和运维成本。  
  
价值排序： **开发者体验 > 生态整合 > 技术创新**。它赢的不是"造了更好的数据库"，而是"把 Postgres 变得像 Firebase 一样好用，同时没牺牲 SQL 生态和可迁移性"。   
  
## 商业模式: 卡 AI 产业链中的重要一环, 然后收"过路费"  
  
Supabase 刚完成 \$500M E 轮，估值 \$10.5B（2026年6月），数据库数量同比增长600%，25万客户只用350人团队支撑，而且 Claude Code 是它今年以来最大的贡献者，AI agent 已经在这个平台上部署了大多数新数据库。这就是收"过路费"最好的基础。  
  
**它在 AI 产业链上卡的是哪个环节？**  
  
不是模型层，不是应用层，是 **"Agent 生成代码 → 需要落地的持久化层"这个衔接点**。  
  
1. **Vibe coding 的默认后端** —— Bolt、Figma、Lovable、Replit 这些 AI 应用生成器都跑在 Supabase 基础设施上。用户对 AI 说"帮我做个App"，AI 落地时默认调用的就是 Supabase。这不是技术选型，是**心智占位** —— 就像你说的"过路费"，位置卡在"意图→实体基础设施"的转换点上。  
  
2. **Agent 记忆与状态的存储层** —— 大部分 Agent 技术栈本来需要单独的向量数据库、单独的认证、单独的文件存储、单独的 API 层、再加一个 Postgres 实例，Supabase 把这些全部合并成一个连接串、一个面板、一个账单。这是"整合税"，本质是帮 Agent 省心智负担。  
  
3. **B2B2C 的乘数效应** —— "Supabase for Platforms" 这个产品让 AI 应用生成器给每个终端客户单独开一个隔离数据库，Supabase 处理多租户，这变成一种完全不同的增长模型 —— 生成器的每个客户都变成 Supabase 的用量。这才是真正的"过路费"结构：它不直接对最终用户收费，而是嵌在每一层转手中抽成。  
  
## 为什么现在 Supabase 暂时很难被替代?  
  
不是技术不可替代（PostgREST、GoTrue 都开源，谁都能抄），而是 **"默认选项"的先发制人的卡位** —— Supabase 现在的位置类似 2006 年的 AWS S3，是 AI Agent 默认假设存在的基础设施层，一旦嵌入开发者工作流形成的复合采纳效应就很难被替换。这是网络效应/路径依赖问题，不是护城河问题。  
  
## 未来被替代的可能性也存在  
  
**平台自建替代**：Anthropic/OpenAI/Cursor 这类"离用户最近的入口"如果自己下场做托管 Postgres（或直接收购一个 Supabase 竞品），Supabase 就从"默认选项"变成"可选项"之一。目前它们选择合作而非自建，但商业逻辑上这是最大威胁。  
  
**Agent 原生数据层出现**：如果有人做出"专为 Agent 设计"而非"通用后端"的存储范式（比如原生支持长期记忆、多 Agent 协作状态同步、无需 SQL 心智负担），会绕开 Supabase 现在"通用 BaaS 硬套在 Agent 场景"的错位 —— 现有评价已经指出 Supabase 是通用后端而非 AI 专属平台，Agent 场景可能还需要自建更多基础设施。 (还有必须指出另一派的观点, 湖库一体 + Agent 数据底座一锅端的产品, 这种产品成熟了, 也没 Supabase 啥事了)   
  
**虚高流量被证伪**：大量 Agent 建的数据库是临时性的（原型、一次性任务），这些数据库计入增长数字但未必转化为付费客户，公司又拒绝披露收入，一旦"过路费"抽不到真实现金流，估值和地位都会被重新定价。  
  
## 总结  
Supabase 现在收的确实是"AI 产业链的过路费" —— 但收的不是技术税，是"心智默认位"税。这种税最脆的地方在于它完全依赖上游（Claude Code、Cursor 等 Agent 平台）继续把它当默认选项，一旦上游想自己收这份过路费（往下游整合），Supabase 的位置随时可能从"收费站"降级为"收费站的一个供应商"。还有一个隐藏对手则是传统数据库厂商“湖库一体 + Agent 数据底座”一锅端的产品, 这种产品成熟了, 也没 Supabase 啥事了, 至少会让 Supabase 的市场大打折扣, 毕竟最有钱的还是大型企业, 老数据库厂商的商务一定不会这么轻易让新产品去打开市场.
