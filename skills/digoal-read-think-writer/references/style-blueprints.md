# 文体蓝图（style-blueprints）

按原文类型挑对应的文体。**不是模板，是判断依据**——写的时候自由组合，不要硬套。

## 速查表

| 原文类型 | 匹配文体 | 钩子策略 | 段落数 | 视觉 |
| --- | --- | --- | --- | --- |
| 行业大事件 / 公司动作 | 大事件体 | 反直觉判断 | 1-2 | 可选 |
| 技术更新 / paper / commit | 知识分享 + 深度洞察 | 数字冲击 / 反问 | 1-3 | 强烈建议 Mermaid |
| 行业争议 / 撕逼 | 吃瓜体 | 私人场景 | 1-2 | 偶尔用对比图 |
| 产品翻车 / 言论离谱 | 吐槽体 | 反问 + 私人反应 | 1 | 通常无图 |
| 人生感悟 / 方法论 | 随想体 | 私人场景 | 1-2 | 通常无图 |
| 战略趋势 / 长期判断 | 深度洞察 | 数字冲击 / 反问 | 1-3 | 可用 mermaid 时序/状态图 |

---

## 1. 大事件体

**场景**：某公司发新品 / 收购 / 财报 / 重大发布。

**骨架（不可见）**：

1. 一句话讲清楚发生了什么
2. 大多数人怎么看（可能错）
3. 德哥怎么看（反直觉点）
4. 第一性原理 + 边界条件
5. 行动建议

**钩子范例**：

> 周三早上我刷到 X 公司发了个 Y。
>
> 第一反应：就这？
>
> 看完细节，我改主意了。
>
> 这事没那么简单。

**节奏**：开头反直觉，中间判断，结尾留给读者想。

---

## 2. 知识分享 + 深度洞察

**场景**：paper / commit / 新 feature / 新工具。

**骨架**：

1. 钩子：用现象引出本质
2. 第一性原理：拆到底层约束
3. 机制：怎么实现的（可配代码/commit）
4. 实操：怎么用、SQL、命令、配置
5. 验证：怎么确认有效
6. 边界：什么场景不灵

**钩子范例**：

> 1000 万行数据，1.5 秒出结果。
>
> 但代价是什么？
>
> 我看完这个 patch 之后，反而不敢用了。

**视觉**：机制图（flowchart / sequenceDiagram）+ 实操流程图。

---

## 3. 吃瓜体

**场景**：公司互怼 / 高管离职 / 行业互喷。

**骨架**：

1. 一句话讲清楚瓜
2. 各方表面立场
3. 德哥判断：到底谁有理，按什么标准
4. 反常识视角：行业里其他人没看到的那一层

**钩子范例**：

> A 公司说 B 抄他。
>
> B 公司说没抄。
>
> 我两边都看了下，发现这事有意思的不是抄没抄。

**注意**：吃瓜体不站队，但可以下判断。下完判断必须能解释机制，不然就是普通吐槽。

---

## 4. 吐槽体

**场景**：产品翻车 / 言论离谱 / 文档稀烂 / 发布会 PPT 跑偏。

**骨架**：

1. 一句话摆事实（具体到细节）
2. 德哥反应（私人口吻）
3. 为什么离谱：拆机制
4. 一句留白

**钩子范例**：

> 这 PPT 我看了三遍。
>
> 第三遍我才确认，我没看错。
>
> 他们真这么写了。

**注意**：吐槽体最容易写成纯情绪输出。**必须有机制解释**——为什么这事离谱，从哪个角度看才有问题。否则就是公众号水文。

---

## 5. 随想体

**场景**：方法论 / 人生感悟 / 行业反思 / 个人经历。

**骨架**：

1. 一个具体的小场景开头（私人）
2. 一句话引出反思
3. 第一性原理
4. 行动建议

**钩子范例**：

> 上周一个朋友问我，PG 还要不要学。
>
> 我愣了一下。
>
> 这个问题，放在 2020 年我会直接回答。
>
> 2026 年，我得想一下。

**注意**：随想体最容易写飘。**每段都必须有"为什么这么想"的具体理由**，不然就是鸡汤。

---

## 6. 深度洞察

**场景**：战略趋势 / 长期判断 / 行业演化 / 技术演进。

**骨架**：

1. 一个数字/事实冲击
2. 反常识判断（先于共识）
3. 第一性原理拆解
4. 案例 / 数据托底
5. 前提 + 反例
6. 行动建议

**钩子范例**：

> 2026 年最被低估的技术，是 X。
>
> 大多数人还在为 Y 吵架。
>
> 我看了一组数据后，改主意了。

**注意**：深度洞察必须有数据/案例托底，否则就是吹牛。**忌空话**。

---

## 文体混合

实操时**常混合**：

- 知识分享 + 深度洞察（最常见，paper/patch 都用这个）
- 大事件 + 深度洞察（公司动作，要带长期判断）
- 吃瓜 + 吐槽（撕逼场景，可以混着用）
- 随想 + 深度洞察（个人反思类）

**判断方式**：如果原文主要是"发生了什么 + 为什么"，就是知识分享+洞察。如果是"谁做了什么 + 对行业意味着什么"，就是大事件+洞察。

## 视觉使用规则

| 场景 | 推荐图表 | 不推荐 |
| --- | --- | --- |
| 解释一个机制 | Mermaid flowchart | 长文字描述 |
| 解释一个调用链 | Mermaid sequenceDiagram | SVG 重画 |
| 对比两套方案 | Mermaid quadrantChart 或表格 | 长段落对比 |
| 状态/迁移 | Mermaid stateDiagram-v2 | 用 flowchart 硬画 |
| 概念地图（抽象） | SVG（存 svg/） | Mermaid 凑合 |
| 评分卡/雷达图 | SVG（简单几何形） | Mermaid（不支持） |

## 自检问题

写完回头看一遍：

1. **文体挑对了吗？** 如果原文是大事件，我用了随想体，会显得软绵绵。
2. **钩子够硬吗？** 反问/数字/反直觉，不达标就重写开头。
3. **判断下了吗？** 还是只描述没立场？
4. **机制讲了吗？** 还是只给结论？
5. **前提说清楚了吗？** 还是一刀切？
6. **视觉帮上忙了吗？** 还是只是装饰？
7. **有活人感吗？** 自己念一遍。
8. **类比是德哥熟的吗？** 不熟就换。

任意一条不过，重写那一段。**不要全篇重写**——精准改。
