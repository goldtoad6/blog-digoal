# 测试矩阵子模板(Test Matrix Template)

> 用于功能设计文档的 §16 验收标准。
> 测试者按本矩阵推导测试用例,**不需要看代码**。
> 矩阵的每一格都必须可执行(有明确的输入/操作/预期)。

---

## 1. 测试维度定义

| 维度 | 取值 | 说明 |
|---|---|---|
| 角色 | 普通用户/VIP/客服/管理员/未登录/系统 | 谁操作 |
| 资源状态 | 初始/进行中/终态/不存在/已禁用 | 操作什么状态的对象 |
| 输入数据 | 合法/非法/边界/特殊字符 | 传什么参数 |
| 操作 | 增/删/改/查/状态变更 | 做什么 |
| 环境 | 正常/弱网/高并发/依赖服务不可用 | 在什么条件下 |

> 矩阵的颗粒度取决于功能复杂度,但每个功能模块至少要覆盖"角色 × 操作"二维。

---

## 2. 通用测试矩阵模板

### 2.1 角色 × 操作矩阵(权限)

| 角色 \\ 操作 | 创建 | 查询(自己) | 查询(他人) | 修改(自己) | 修改(他人) | 删除 | 状态变更 |
|---|---|---|---|---|---|---|---|
| 普通用户 | ✓ | ✓ | ✗ 403/404 | ✓ (限定字段) | ✗ | ✗ | ✓ (限定类型) |
| VIP 用户 | ✓ | ✓ | ✗ | ✓ (更多字段) | ✗ | ✗ | ✓ (限定类型) |
| 客服 | ✗ | ✓ (脱敏) | ✓ (脱敏) | ✗ | ✗ | ✗ | ✓ (限定类型) |
| 管理员 | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| 未登录 | ✗ 401 | ✗ 401 | ✗ 401 | ✗ 401 | ✗ 401 | ✗ 401 | ✗ 401 |
| 系统服务 | ✓ | ✓ | ✓ | ✓ | ✓ | ✗ | ✓ |

**要求**:
- 每格必须有"✓ / ✗ / 限定",不能写"看情况"。
- ✗ 必须给出错误码(403 / 401 / 404)。
- "限定" 必须说明限定条件。

### 2.2 资源状态 × 操作矩阵

| 资源状态 \\ 操作 | 编辑 | 删除 | 状态变更 A | 状态变更 B |
|---|---|---|---|---|
| 初始 | ✓ | ✓ | ✓ | ✗ |
| 进行中 | ✗ | ✗ | ✓ | ✗ |
| 终态-成功 | ✗ | ✗ | ✗ | ✗ |
| 终态-失败 | ✗ | ✗ | ✗ | ✗ |
| 不存在 | ✗ 404 | ✗ 404 | ✗ 404 | ✗ 404 |
| 已禁用 | ✗ | ✗ | ✗ | ✗ |

**与 §6 状态机对应**:每个状态 × 操作交叉都必须在状态机里能找到定义。

### 2.3 输入数据矩阵

| 字段 | 取值 | 期望行为 |
|---|---|---|
| 字符串-空 | "" | 拒绝,error="field required" |
| 字符串-正常 | "hello" | 通过 |
| 字符串-超长 | 256+ 字符 | 拒绝,error="max length 255" |
| 字符串-Unicode | "中文 🚀" | 通过(允许时) |
| 字符串-注入 | "' OR 1=1;--" | 拒绝(参数校验)或安全处理 |
| 字符串-前后空格 | " hello " | trim 后接受,或拒绝(看设计) |
| 数值-0 | 0 | 拒绝,error="min 1" |
| 数值-负 | -1 | 拒绝 |
| 数值-最大边界 | 2147483647 | 接受或拒绝(看设计) |
| 数值-溢出 | 2147483648 | 拒绝(类型不匹配) |
| 数组-空 | [] | 拒绝(必填且 ≥1) |
| 数组-单元素 | [x] | 通过 |
| 数组-最大边界 | [x]*50 | 通过 |
| 数组-超长 | [x]*51 | 拒绝 |
| 枚举-合法 | "v1" | 通过 |
| 枚举-非法 | "v9" | 拒绝 |
| 时间-过去 | "2020-01-01" | 看设计 |
| 时间-未来 | "2099-01-01" | 看设计 |
| 时间-格式错误 | "2020/01/01" | 拒绝 |

## 3. 用例规格

> 每条用例必须包含:前置、操作、断言(逐字段)、清理。

### 3.1 用例 ID 规范

`<模块>-<类型><编号>`

- 模块:ORD(订单)/USR(用户)/PAY(支付)
- 类型:N(正常)/A(异常)/P(权限)/S(状态机)/C(并发)/E(环境)
- 编号:3 位数字

例:ORD-N001、PAY-A023

### 3.2 用例规格模板

| 字段 | 内容 |
|---|---|
| 用例 ID | ORD-N001 |
| 标题 | 普通用户成功创建一个订单 |
| 优先级 | P0 |
| 关联需求 | PRD §3.1 / 设计 §7.2.1 |
| 前置条件 | 1. 用户已登录(uid=1001)<br>2. 商品 1001 状态=在售,库存=100<br>3. 用户地址 8888 有效 |
| 操作步骤 | 1. POST /api/orders<br>Body: `{"items":[{"product_id":1001,"quantity":2}],"address_id":8888}` |
| 预期结果 | 1. HTTP 200<br>2. 响应 order_id > 0<br>3. 响应 status = 0(待支付)<br>4. 响应 total_amount 与商品单价×2 一致(精度 0.01)<br>5. 响应 created_at 为当前时间 ±5s<br>6. 数据库 orders 表新增一条记录<br>7. 数据库 order_items 表新增对应记录<br>8. 库存从 100 减为 98 |
| 后置条件 | 清理:删除测试订单、恢复库存 |
| 自动化 | 是(API 测试)/ 否(人工) |

**断言必须能逐字段写**。"返回成功"不算断言。

## 4. 必测用例清单(完整文档必须覆盖)

| 类别 | 必测内容 | 数量参考 |
|---|---|---|
| 正常路径 | 核心功能 happy path | ≥ 5 条/模块 |
| 异常输入 | 参数校验全覆盖 | 字段数 × 5 |
| 异常业务 | 资源不存在/已禁用/越权 | ≥ 10 条/模块 |
| 状态机 | 每个合法/非法转移 | 见状态机矩阵 |
| 权限 | 角色 × 操作矩阵 | 见 §2.1 |
| 性能 | 关键接口的 p99 / 错误率 | ≥ 3 条/模块 |
| 并发 | 双写、竞态、重入 | ≥ 2 条/模块 |
| 兼容性 | 旧版本客户端/服务端 | 视情况 |
| 安全 | SQL 注入、XSS、越权、敏感数据 | ≥ 5 条/模块 |
| 灾备 | 依赖服务不可用 | ≥ 1 条/依赖 |

## 5. 性能与稳定性测试规格

### 5.1 性能基线

| 接口 | 场景 | p50 | p99 | 错误率 | 备注 |
|---|---|---|---|---|---|
| POST /api/orders | 单商品 | < 50ms | < 200ms | < 0.1% |  |
| POST /api/orders | 50 商品 | < 200ms | < 500ms | < 0.1% |  |
| GET /api/orders/{id} |  | < 20ms | < 100ms | < 0.1% |  |

### 5.2 容量测试

| 维度 | 目标 | 测试方式 |
|---|---|---|
| 日订单量 | 100w | 压测 24h |
| 峰值 QPS | 10w | 阶梯压测 |
| 历史订单 | 10 亿 | 兼容性测试 |

### 5.3 稳定性

| 场景 | 持续时间 | 预期 |
|---|---|---|
| 持续峰值 | 4h | 错误率不上升,无内存泄漏 |
| 浪涌 | 0→10w QPS 瞬间 | 5s 内恢复,无 5xx 雪崩 |
| 依赖抖动 | 30s 不可用 | 降级,主流程可用 |

### 5.4 故障注入

| 故障 | 预期 |
|---|---|
| 数据库主从切换 | 30s 内恢复,期间请求重试 |
| 缓存服务宕机 | 走数据库,降级但功能可用 |
| 消息队列堆积 | 不阻塞主流程,异步消化 |

## 8. 验收签收标准

- [ ] 必测用例 100% 执行

- [ ] P2 用例 ≥ 95% 通过
- [ ] 性能基线达成
- [ ] 安全扫描无高危
