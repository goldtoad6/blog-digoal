# 模块精度档位判断标准

这份参考文件帮助你判断每个模块需要哪些"按需"文档元素。核心原则只有一句话：**接口字段级精度是所有模块的底线,不会因为模块"简单"而降低;三个判断只决定要不要额外加状态图、数据模型表、异常场景表。**

## 判断一：是否需要数据模型/字段定义表

**问**：这个模块的数据需要持久化存储吗(建表、写数据库、缓存关键状态)?

- **是** → 需要独立的数据模型表,字段包括:字段名、类型、是否必填、默认值、约束规则(唯一性/长度/格式/取值范围)、说明。这张表独立于接口请求/响应之外存在,因为同一个数据模型可能被多个接口复用(创建接口用一部分字段,查询接口返回另一部分字段),把模型单独定义一次,比在每个接口里重复写更不容易产生字段定义不一致的问题。
- **否**(纯计算型、纯转发调用型、无状态聚合型模块) → 不需要单独的数据模型表,字段定义直接体现在接口请求/响应里就够了,不要为了"看起来完整"硬造一张表。

**判断时的常见误区**：不要把"接口的请求参数"和"持久化的数据模型"混为一谈。一个查询接口可能有 10 个请求参数,但这个模块本身不存任何数据(纯转发查询给下游服务)——这种情况下接口字段表要写,但数据模型表不需要。反过来,一个模块即使对外只暴露 2 个简单接口,但背后存了一张有 15 个字段的表,数据模型表也必须写,因为测试者需要知道这些字段的约束才能设计数据相关的边界用例(比如某字段唯一索引,测试者需要测重复插入的报错)。

## 判断二：是否需要状态转换图

**问**：这个模块里的核心实体,是否有"多个明确状态 + 状态之间存在转换规则"的特征?

典型信号：
- 实体的生命周期里有"创建→处理中→完成/失败"这种阶段性变化(订单、工单、审批、任务)
- 不同状态下允许的操作不同(比如"已发货"的订单不能再修改地址)
- 存在状态会因为超时、人工干预、系统事件而自动迁移

- **是** → 必须画状态转换图(Mermaid `stateDiagram-v2` 最合适,因为它专门表达状态机)。图里要包含:所有合法状态、每个转换的触发条件、是否存在自动超时迁移。文字层面还要补一张"非法转换清单"——明确写出哪些转换是不允许的(比如"已取消"不能转回"待支付"),因为这正是测试者会拿来设计反例用例的地方,图本身通常只画合法路径,容易让人误以为"图里没画的就是无所谓",所以必须用文字单独强调非法转换。
- **否**(普通 CRUD,创建后状态基本不变,或者只有"启用/禁用"这种单一布尔开关) → 不需要强行画状态图。一个只有"启用/停用"两态、且没有复杂触发条件的模块,画一张状态图反而是噪音,文字说明"该资源仅有启用/停用两态,可由管理员任意切换,无中间状态"就足够清楚。

## 判断三：是否需要异常/边界场景表

**问**：这个模块是核心路径还是辅助功能?出错的影响面有多大?

判断"核心/高风险"的参考信号(满足任意一条即算核心)：
- 主流程必经,功能不可用会导致用户完全无法完成关键任务(比如下单、支付、登录)
- 涉及金钱、权限、数据一致性等高代价错误的场景
- 涉及外部依赖(第三方支付、短信、第三方 API),外部依赖的不稳定性会传导成本模块的故障
- PRD 里这个功能被标记为 Must Have 且是 P0 优先级

- **是核心/高风险** → 必须列异常/边界场景表,至少覆盖:空值/缺失参数、输入超出范围、并发操作冲突、权限不足、第三方依赖超时或返回异常、重复提交。每一行写清楚"触发条件→期望系统行为→对应的错误码或界面提示",这样测试者可以直接照着设计反例用例,不用自己猜系统应该怎么响应。
- **是辅助功能**(比如一个"意见反馈"提交入口,出错了用户重试一下就行,不影响其他功能) → 可以把异常处理合并写进验收标准的字段说明里,不需要单独成表。比如直接在接口的字段说明里写一句"content 为空时返回 40001 参数错误",而不是专门开一个章节列异常场景表。

## 三个判断之间的关系

三个判断互相独立,同一个模块可能是"是、否、是"的任意组合,不要假设它们总是同时成立或同时不成立。常见组合举例:

- **订单模块**：有持久化(是)+ 有状态机(是)+ 核心路径(是)→ 三样都要,这是最重的一类模块。
- **意见反馈模块**：有持久化(是,反馈内容要存)+ 无状态机(否,提交后就是终态)+ 辅助功能(否,不需要异常表)→ 只需要数据模型表 + 接口契约,不需要状态图和异常表。
- **价格计算模块**：无持久化(否,纯计算,输入价格返回折后价)+ 无状态机(否)+ 核心路径(是,如果算错价格影响范围很大)→ 不需要数据模型表和状态图,但异常场景表必须有(比如输入负数价格、折扣率超过 100% 等边界要测)。
- **第三方地图服务转发模块**：无持久化(否,纯转发)+ 无状态机(否)+ 视依赖重要性而定(如果地图服务挂了导致核心下单流程无法继续,算核心;如果只是个"附近门店展示"的辅助信息,算辅助)→ 视具体业务影响判断,不要机械地认为"调用第三方"就一定是核心或一定是辅助。

把这套判断标准应用到每个模块时,最终目的是省下"无意义的重复劳动"(给一个没有状态机的简单开关模块画状态图),而不是省下"必要的精度"(任何模块的接口字段级描述都不能省)。如果拿不准某个模块该归哪一类,倾向于按"更高规格"处理——多写一张表的成本,远低于测试者因为文档缺失而漏测一个真实存在的高风险场景的成本。
