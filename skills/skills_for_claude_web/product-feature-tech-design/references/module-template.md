# 功能设计文档完整模板

这是整份文档的结构骨架。每个模块章节按 `references/precision-rules.md` 的判断结果,选择性包含"按需"小节,但"必选"小节任何模块都不能省。

## 文档头部

```markdown
# [产品名称] 功能设计文档

> 本文档基于 [PRD 文件名/来源] 编写,面向开发、测试、Review 三方协作使用。三方互相隔离,本文档是唯一的对齐依据。

**文档类型**：功能设计文档（Functional & Technical Design Doc）
**关联 PRD**：[PRD 文件名或链接]
**文档状态**：草稿 / 评审中 / 已批准
**版本**：v1.0
**作者**：[架构师/作者]
**最后更新**：[日期]
**模块清单**：[模块A]、[模块B]、[模块C]...
```

## 一、整体架构

### 1.1 模块依赖关系图

用 Mermaid 画跨模块的调用/依赖关系,例如：

```markdown
\`\`\`mermaid
graph TD
    A[订单模块] -->|创建订单后调用| B[库存模块]
    A -->|支付成功后调用| C[通知模块]
    B -->|库存不足时调用| C
    A -.异步消息.-> D[数据分析模块]
\`\`\`
```

图例说明放在图下方,用文字说明箭头含义(同步调用/异步消息/数据依赖等),不要假设读图的人能猜出箭头类型。

### 1.2 模块概览表

| 模块 | 核心职责 | 是否核心路径 | 是否涉及状态机 | 是否涉及持久化 |
|------|---------|------------|--------------|--------------|
| [模块A] | [一句话说明] | 是/否 | 是/否 | 是/否 |

这张表是给三方协作者的"导航索引"——reviewer 想知道哪些模块该重点看,测试者想知道哪些模块该多设计反例用例,扫一眼这张表就知道往下该重点读哪几个章节。

## 二、模块详细设计（每个模块重复以下结构）

### 2.X [模块名称]

#### 2.X.1 模块说明（必选）

一句话说明这个模块解决什么问题,对应 PRD 里的哪些功能点(Must/Should Have 编号或名称,方便追溯)。

#### 2.X.2 数据模型（按需——仅当模块有持久化存储时包含）

| 字段名 | 类型 | 是否必填 | 默认值 | 约束规则 | 说明 |
|--------|------|---------|--------|---------|------|
| id | string | 是 | 系统生成 | UUID 格式 | 主键 |
| [字段] | [类型] | 是/否 | [默认值或"无"] | [长度/格式/范围/唯一性] | [业务含义] |

如果存在字段间的联动约束(比如"当 status=已完成 时, completed_at 必须有值"),在表格下方用文字单独说明,不要指望表格本身能表达跨字段规则。

#### 2.X.3 状态转换（按需——仅当模块涉及明显状态机时包含）

```markdown
\`\`\`mermaid
stateDiagram-v2
    [*] --> 待支付: 创建订单
    待支付 --> 已支付: 支付成功
    待支付 --> 已取消: 超时未支付(30分钟)/用户主动取消
    已支付 --> 已发货: 仓库确认发货
    已发货 --> 已完成: 用户确认收货/7天自动确认
    已支付 --> 退款中: 用户申请退款
    退款中 --> 已退款: 退款成功
\`\`\`
```

**非法转换清单**(图中不会画出来,但必须用文字明确禁止)：

| 起始状态 | 禁止转向 | 原因 |
|---------|---------|------|
| 已取消 | 任何状态 | 终态,不可逆 |
| 已发货 | 待支付 | 已完成支付确认,不可回退 |

#### 2.X.4 接口契约（必选——每个接口都要写到字段级）

##### 接口：[接口名称,如"创建订单"]

- **请求方式与路径**：`POST /api/v1/orders`（如果尚未确定真实路由,标注"待开发阶段确认实际路由,此处为语义化占位路径"）

**请求参数**：

| 参数名 | 类型 | 是否必填 | 默认值 | 约束规则 | 说明 |
|--------|------|---------|--------|---------|------|
| user_id | string | 是 | 无 | 必须是已登录用户的合法 ID | 下单用户 |
| items | array | 是 | 无 | 数组长度 1-50 | 订单商品列表 |
| items[].sku_id | string | 是 | 无 | 必须是存在且在售的 SKU | 商品 ID |
| items[].quantity | int | 是 | 无 | 1-999 之间的正整数 | 购买数量 |

**响应结构（成功）**：

```json
{
  "order_id": "string，订单唯一标识",
  "status": "string，订单状态，取值见 2.X.3 状态转换",
  "total_amount": "number，订单总金额，单位：元，保留两位小数",
  "created_at": "string，ISO8601 时间格式"
}
```

**错误码**：

| 错误码 | 触发场景 | 错误信息示例 |
|--------|---------|------------|
| 40001 | 必填参数缺失或格式不对 | "参数 items 不能为空" |
| 40301 | 用户未登录或登录态过期 | "请先登录" |
| 40401 | sku_id 对应商品不存在或已下架 | "商品不存在或已下架" |
| 40901 | 库存不足导致下单失败 | "库存不足，剩余 X 件" |
| 50001 | 系统内部错误（数据库写入失败等） | "系统繁忙，请稍后重试" |

**幂等性/并发说明**：[比如:"同一用户 5 秒内重复提交相同商品组合的请求,只生效第一次,后续返回已创建的订单 ID,通过 user_id + items 哈希做幂等键"。如果该接口不存在幂等/并发风险,直接写"该接口无重复提交风险,因为xxx"]

(对模块内的每个接口重复上面这套结构)

#### 2.X.5 异常与边界场景（按需——仅当模块为核心/高风险路径时包含独立表格,辅助模块可并入字段说明）

| 场景 | 触发条件 | 期望系统行为 | 对应错误码/提示 |
|------|---------|------------|---------------|
| 并发超卖 | 同一商品库存仅剩1件,两个请求同时下单 | 仅一个请求成功,另一个返回库存不足 | 40901 |
| 第三方支付超时 | 支付网关 5 秒未响应 | 订单状态保持"待支付",不创建重复扣款记录,提示用户稍后查询结果 | 50002 |
| 权限越权 | 用户尝试查看不属于自己的订单 | 拒绝访问,不暴露该订单是否存在 | 40301 |

#### 2.X.6 模块验收标准（必选）

用 Given/When/Then 把这个模块最关键的几条验收规则汇总一遍(不是重复前面所有细节,是挑出"如果这条不满足,这个模块就不能算完成"的核心规则)：

- Given [前置条件] When [操作] Then [期望结果,包含具体数值/字段]
