# PostgreSQL 插件开发场景：测试环境搭建参考

数据库插件这类开发任务的特殊之处在于：测试不是简单跑一下函数，而是需要先有一个"活的"数据库
实例，插件加载进去之后，再通过 SQL 或者插件提供的接口去验证行为。这份参考列出这类场景下，
搭建测试环境通常需要走的步骤和常见坑点，具体命令请结合实际的 PostgreSQL 版本和插件构建方式
（Makefile / PGXS / meson 等）调整，不要照抄下面的示例当成万能脚本。

## 需要提前向用户确认的信息

进入测试阶段之前，应该已经从用户那里拿到（如果在"收集必要输入"阶段没问到，测试阶段发现缺了
要回头去问，不要自己瞎猜，猜错了环境搭不起来，浪费的是后面所有轮次的时间）：

- PostgreSQL 源码路径，或者是否已经有编译好的 PostgreSQL 安装
- 目标 PostgreSQL 大版本（插件的 API 经常和大版本绑定，搭错版本会出现编译都通不过的情况）
- 是用全新搭建一套测试集群，还是连接到用户已经准备好的实例
- 如果是新搭建：用什么端口、数据目录放哪、需要哪些非默认的 `postgresql.conf` 参数
  （插件经常需要 `shared_preload_libraries` 之类的参数才能正常加载）
- 操作系统/容器环境里是否已经装好了编译依赖（`make`、对应版本的 `postgresql-server-dev`
  或等价的头文件包等）

## 典型流程

1. **编译插件**：进入插件源码目录，按其构建方式编译（最常见的是 PGXS：`make`，
   依赖 `pg_config` 能在 PATH 里找到，且 `pg_config` 对应的版本要和目标 PostgreSQL 版本一致——
   版本不匹配是这一步最常见的失败原因）。
2. **初始化测试集群**（如果不是连接到已有实例）：用 `initdb` 在指定数据目录初始化一个全新集群，
   注意这一步通常需要指定字符集/locale，避免用了和后续测试断言不兼容的本地化设置。
3. **配置必要参数**：如果插件需要在启动时加载（比如通过 `shared_preload_libraries`），
   要在启动数据库*之前*改好 `postgresql.conf`，不能等数据库起来了再补——这类参数通常需要重启
   才生效，先改对了再启动能省一轮无意义的重启。
4. **启动数据库**：用 `pg_ctl start` 或者发行版对应的方式拉起来，启动后检查日志确认没有报错
   （插件加载失败、参数不兼容等问题通常第一时间就在启动日志里，不要跳过这一步直接假设启动成功）。
5. **安装/初始化插件**：连上数据库执行 `CREATE EXTENSION` 或者对应插件文档要求的初始化语句，
   确认插件的 SQL 对象（函数、类型、视图等）确实创建出来了。
6. **执行测试**：可以是手写的 SQL 脚本配合预期输出比对，也可以是 PostgreSQL 生态常用的
   `pg_regress` 风格测试（`.sql` 输入 + `.out` 期望输出），具体形式跟着插件本身已有的测试约定走，
   没有约定的话选一种团队/项目里常见的方式即可，不必另起一套。
7. **清理**：测试结束后，如果是本次新搭建的临时集群，跑完测试后停掉数据库
   （`pg_ctl stop`），并视情况决定是否保留数据目录方便排查问题，还是清理掉。

## 常见坑点

- **版本不匹配**：`pg_config` 指向的版本和插件代码假设的版本不一致，编译能过但运行时行为不对，
  或者直接编译失败——报错信息里如果提到找不到某个符号/某个头文件，先怀疑版本问题。
- **`shared_preload_libraries` 漏配**：有些插件必须在这个参数里声明才能正常工作，
  仅仅 `CREATE EXTENSION` 而不配置这个参数，看起来"装上了"但实际功能不工作，容易被误判为
  代码 bug，其实是环境没配对。
- **权限问题**：测试集群的数据目录权限、运行数据库进程的用户身份，如果跟环境里其他已有的
  PostgreSQL 安装权限冲突，会出现启动失败但报错信息不直观的情况。
- **端口冲突**：如果环境里可能已经有一个 PostgreSQL 实例在跑，新建的测试集群要用不同端口，
  否则启动会因为端口被占用而失败，报错有时候会被误读成别的问题。
- **测试之间没有清理干净状态**：如果测试用例之间共享同一个数据库/集群，前一条用例留下的数据
  可能影响后一条用例的判断，测试用例设计阶段就要考虑好这一点（比如每条用例自己创建和清理用到的
  schema/表）。

## 报告里应该体现的信息

无论环境搭建顺利还是遇到了问题，测试报告里都应该说清楚：用的是哪个 PostgreSQL 版本、
新搭建的还是连接已有实例、有没有用到非默认配置参数（用了哪些）、启动/初始化过程中是否有
报错或警告（即使最终成功了，警告也值得记一笔，免得下一轮重新搭建时再踩一次）。
